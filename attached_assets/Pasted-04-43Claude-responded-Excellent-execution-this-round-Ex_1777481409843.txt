2. forgot-password — randomBytes is the wrong cost to equalise. randomBytes(32) is microseconds. The cost difference between the two paths is the DB write (createPasswordResetToken) plus the email send. You moved the token generation to both paths, but the DB write and email still only happen on the user-found path. Net timing tell: roughly the cost of an INSERT plus a network round-trip to your SMTP — easily 50-200ms.
This is a harder one to equalise without doing real work for fake users. The cleanest pattern is to push email-send onto a queue/worker and respond immediately on both paths after just the DB lookup:
tsconst user = await storage.getUserByEmail(email);
if (user) {
  await emailQueue.enqueue('password-reset', { userId: user.id, ... });
}
return res.json({ message: 'If an account with this email exists...' });
Both paths do exactly one DB read and respond. The actual token generation, DB write, and email send happen out-of-band. This also fixes the unrelated DoS where someone hammers forgot-password with real emails to generate token churn.
If you don't have a queue worker yet, the interim is to fire-and-forget the email work without awaiting it — set immediate response, do the rest in background:

Slightly less clean (errors logged not surfaced), but fixes the timing tell with no infra change. Just be careful that any thrown error in the background doesn't crash the process — that .catch matters.
